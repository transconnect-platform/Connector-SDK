= Connector SDK Architecture

== Introduction

The TRANSCONNECT Connector SDK provides a standardized framework for integrating external systems with the TRANSCONNECT platform. This document outlines the architecture, core components, and interaction patterns of the SDK.

=== Key Concepts

The SDK is built around two primary connector types:

[cols="1,4", options="header"]
|===
|Connector Type |Purpose
|*Consumer Connectors* |Receive and process messages from TRANSCONNECT, interacting with external systems to fulfill requests
|*Producer Connectors* |Monitor external systems and generate messages that are sent to TRANSCONNECT
|===

Both connector types share a common design philosophy:

* Clear separation between connector and connection
* Standardized message handling
* Configuration-driven behavior
* Lifecycle management

== Best Practices

When implementing connectors with the SDK, follow these patterns:

1. **Separation of Concerns**
* Connectors create connections and provide metadata
* Connections contain business logic and handle message processing

2. **Error Handling**
* Use try-with-resources for proper resource cleanup
* Wrap external exceptions in link:../_attachments/javadoc/io/transconnect/connector/api/TransconnectConnectorException.html[`TransconnectConnectorException`,window=_blank]
* Log meaningful error messages

3. **Configuration Management**
* Define clear property names with helpful descriptions
* Provide sensible defaults where possible
* Validate configuration early

4. **Connection Lifecycle**
* Ensure proper cleanup in the `close()` method
* Implement thread-safe connection state checking
* Design connections to be stateless where possible
